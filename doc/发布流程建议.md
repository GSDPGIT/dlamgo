# 发布流程建议

本文档用于说明当前仓库推荐的版本发布方式。

## 一、当前发布体系

当前仓库已经拆分成三条工作流：

- `docker-build.yml`
  用于持续集成校验

- `release.yml`
  用于 tag 发布与 GitHub Release 附件生成

- `docker-publish.yml`
  用于 Docker Hub 镜像发布

这样拆分的好处是：

- 日常提交只跑校验，不污染发布流程
- 发布附件与镜像推送分离，职责更清晰
- 可以按需启用 Docker Hub 发布

---

## 二、推荐发布节奏

建议使用以下节奏：

1. 本地完成开发与联调
2. 更新文档与变更说明
3. 推送到 `main`
4. 等 CI 工作流通过
5. 推送标签，例如 `v1.0.0`
6. 等 `release.yml` 自动创建 GitHub Release
7. 如果配置了 Docker Hub secrets，再由 `docker-publish.yml` 自动推送镜像

---

## 三、推荐标签规范

建议统一使用：

- `v主版本.次版本.修订版本`

例如：

- `v0.1.0`
- `v0.1.1`
- `v0.2.0`

优点：

- 能直接匹配当前工作流的触发规则
- 版本含义清晰
- 便于自动比较上一个标签与当前标签的提交区间

不建议使用：

- `release-1`
- `test`
- `final`

---

## 四、Release 工作流会做什么

当前 `release.yml` 在你推送 `v*` 标签时会自动：

- 构建 Go 后端发布二进制
- 构建前端静态产物
- 校验 Docker 镜像可构建
- 自动创建 GitHub Release
- 自动生成版本说明正文
- 自动生成标签对应的变更摘要文件
- 自动上传发布附件

---

## 五、当前 Release 默认附件

当前 `release.yml` 默认会上传：

- Linux AMD64 后端二进制包
- Linux ARM64 后端二进制包
- Windows AMD64 后端二进制包
- 前端 `dist` 构建包
- `docker-compose.yml`
- `.env.example`
- `install-node.sh`
- `README.md`
- `DOCKER.md`
- `CHANGELOG.md`
- `发布流程建议.md`
- `仓库结构说明.md`
- `VERSION.txt`
- `RELEASE_NOTES.md`
- `CHANGELOG-v版本号.md`
- `RELEASE_BODY.md`
- `SHA256SUMS.txt`

---

## 六、Docker Hub 发布说明

当前仓库已经提供：

- `.github/workflows/docker-publish.yml`

如果你要启用 Docker Hub 自动发布，请配置仓库 Secrets：

- `DOCKERHUB_USERNAME`
- `DOCKERHUB_TOKEN`

当前 `docker-publish.yml` 默认会发布：

- `linux/amd64`
- `linux/arm64`

并自动生成：

- `latest`
- 当前标签名

两个镜像标签。

---

## 七、建议的发布前检查清单

发布前建议确认：

- Go 后端本地可编译
- 前端本地可构建
- Docker `backend` 镜像可构建
- Docker `frontend` 镜像可构建
- `docker compose up -d` 可正常启动
- 管理员登录接口正常
- README 与实际部署方式一致
- `.env.example` 参数说明完整
- 节点安装脚本路径已正确

---

## 八、建议写清楚的发布内容

建议在每次发布时明确说明：

- 本次版本号
- 新增功能
- 修复内容
- 升级注意事项
- 是否涉及数据库迁移
- 是否涉及环境变量变更
- 是否涉及节点安装方式变更
