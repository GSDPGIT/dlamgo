# 发布流程建议

本文档用于说明当前仓库推荐的版本发布方式。

## 一、当前现状

当前仓库已经具备：

- Go 后端本地构建能力
- 前端构建能力
- Docker 镜像本地构建能力
- `docker-compose.yml` 作为默认部署入口
- GitHub Actions 自动构建校验流程

当前仓库当前已经拆分成两条工作流：

- `docker-build.yml`
  用于持续集成校验

- `release.yml`
  用于 tag 发布

其中 CI 默认只做：

- Go 后端编译检查
- 前端构建检查
- Docker 镜像可构建性检查

它不会默认自动推送 Docker 镜像。

当前 `release.yml` 会在你推送 `v*` 标签时自动：

- 构建 Go 后端发布二进制
- 构建前端静态产物
- 校验 Docker 镜像可构建
- 自动创建 GitHub Release
- 上传发布附件
- 自动生成 `VERSION.txt`
- 自动生成当前标签对应的版本说明文件
- 自动生成当前标签对应的变更摘要文件

## 二、推荐发布节奏

建议使用以下节奏：

1. 在本地完成开发和联调
2. 更新 `README.md` / `DOCKER.md` / 变更说明
3. 推送到 `main`
4. 等 GitHub Actions 通过
5. 推送标签，例如 `v1.0.0`
6. 等 `release.yml` 自动创建 GitHub Release
7. 如有需要，再单独推送 Docker 镜像

## 三、推荐版本号格式

建议使用语义化版本：

- `v1.0.0`
- `v1.0.1`
- `v1.1.0`

含义建议：

- `主版本`
  有明显不兼容变更时增加

- `次版本`
  有新功能但兼容旧配置时增加

- `修订版本`
  修复 bug 或文档更新时增加

## 四、建议的发布前检查清单

发布前建议确认：

- Go 后端本地可编译
- 前端本地可构建
- Docker `backend` 镜像可构建
- Docker `frontend` 镜像可构建
- `docker compose up -d` 可正常启动
- 管理员登录接口正常
- README 与实际部署方式一致
- `.env.example` 参数说明完整

## 五、当前 Release 附件内容

当前 `release.yml` 默认会上传：

- Linux AMD64 后端二进制包
- Linux ARM64 后端二进制包
- Windows AMD64 后端二进制包
- 前端 `dist` 构建包
- `docker-compose.yml`
- `.env.example`
- `DOCKER.md`
- `README.md`
- `CHANGELOG.md`
- `VERSION.txt`
- `RELEASE_NOTES.md`
- `CHANGELOG-v版本号.md`
- `SHA256SUMS.txt`

## 六、GitHub Release 建议内容

建议在每次发布时写清楚：

- 本次版本号
- 新增功能
- 修复内容
- 升级注意事项
- 是否涉及数据库迁移
- 是否涉及环境变量变更

## 七、如果需要自动推送 Docker 镜像

如果后续你希望 GitHub Actions 自动推送 Docker 镜像，可以在仓库中新增以下 secrets：

- `DOCKERHUB_USERNAME`
- `DOCKERHUB_TOKEN`

然后再新增单独的“发布工作流”，按 tag 触发推送。

不建议直接把“校验工作流”和“发布工作流”混成一个。

建议继续保持拆分：

- `docker-build.yml`
  只做构建、测试、校验

- `release.yml`
  只做 tag 发布与 Release 附件上传

- `docker-publish.yml`
  如后续需要，再单独负责镜像推送

## 八、推荐标签命名规范

建议统一使用：

- `v主版本.次版本.修订版本`

例如：

- `v0.1.0`
- `v0.1.1`
- `v0.2.0`

优点：

- 能直接匹配当前 `release.yml` 的触发规则
- 版本含义清晰
- 便于后续自动比较上一个标签与当前标签的提交区间
